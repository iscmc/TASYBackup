1. Visao Geral

O TASYBackup e uma aplicação PHP em estilo MVC simples, hospedada no IIS, utilizada para copiar dados clinicos e cadastrais do banco Oracle de producao do TASY para um banco Oracle XE local. O objetivo e sustentar um ambiente de contingência hospitalar offline/local, consumido principalmente pelo portal ISCMC.

Origem

Oracle 19C do TASY, acessado por OCI8 a partir do PHP.

Destino

Oracle 21C XE local no mesmo servidor Windows.

Acesso operacional

Dashboard web manual e servico Windows automatizado.

Escopo atual

Sincronização de dados + controle da tabela CONFIG para contingência.

2. Objetivo de Negócio

O sistema atende tres objetivos principais:

  1. Disponibilizar no Oracle XE local um espelho operacional dos dados críticos das últimas 72 horas.
  2. Manter historico acumulado completo de tabelas mestre essenciais ao contexto assistencial e administrativo.
  3. Permitir que o portal ISCMC continue funcionando localmente em contingência, inclusive sem dependencia de internet.

Relação com o portal ISCMC

O portal ISCMC, hospedado em outra pasta do mesmo IIS, consome o mesmo schema SYSTEM do Oracle XE local. Por isso, a consistencia do TASYBackup impacta diretamente a disponibilidade das informacoes criticas vistas pelos usuários no Portal de Contingência.

Caracteristicas arquiteturais

  • Fluxo manual e fluxo automático compartilham o mesmo nucleo funcional de sincronização.

5. Bancos de Dados e Tabelas de Controle

Bancos utilizados

  • Banco de origem: Oracle do TASY, schema principal configurado como TASY.
  • Banco de destino: Oracle XE local, com schema operacional utilizado pelo sistema e pelo portal.

Tabelas de controle esperadas no Oracle XE

Tabela Finalidade
TASY_SYNC_CONTROL Controla ultimo sync por tabela, total local e metadata minima do processo.
TASY_SYNC_LOGS Armazena registros resumidos das sincronizações executadas.
CONFIG Armazena sinalizadores de contingência, inclusive FRONTEND_ACTIVE.

Premissas importantes

  • As tabelas de negócio de destino ja existem no XE, com a mesma estrutura da origem.
  • O projeto não cria tabelas de negócio; ele apenas insere e atualiza.
  • As chaves fisicas entre PESSOA_FISICA e COMPL_PESSOA_FISICA foram desativadas no XE por performance.

6. Politica de Sincronização por Tabela

A politica de sincronização esta centralizada em app/config/database.php. Cada tabela possui metadados como control_column, key_column, key_columns, sync_hours, retention_mode, sync_priority e, quando necessario, batch_size.

Tabelas com historico acumulado completo

Tabela Motivo Modo
MEDICOCadastro mestrefull
PESSOA_FISICACadastro mestre basefull
COMPL_PESSOA_FISICAComplemento cadastral com chave compostafull
SETOR_ATENDIMENTODimensao organizacionalfull
UNIDADE_ATENDIMENTODimensao operacionalfull
USUARIOIdentidade local e autenticaçãofull

Tabelas com janela movel de 72 horas

As demais tabelas configuradas trabalham em modo rolling, ou seja, sincronizam os registros alterados dentro da janela de 72 horas e aplicam purge local dos dados anteriores a esse intervalo.

Ordem de sincronização

A ordem e definida por sync_priority. A sincronização prioriza cadastros mestre antes das tabelas transacionais, reduzindo riscos de dependencia logica.

7. Fluxo Manual via Dashboard

Fluxo atual correto

index.php
  -> app/bootstrap.php
    -> HomeController
      -> SyncModel::forceTableSync()
        -> BackupModel::insertDataToLocal()
          -> fetch / stream / batch
          -> upsert local
          -> retention policy
          -> update TASY_SYNC_CONTROL
          -> logSyncToDatabase()

Comportamento operacional

  • O dashboard permite disparar sincronização por tabela ou sincronização completa.
  • O metodo forceSync() do HomeController utiliza o fluxo estabilizado do SyncModel.
  • O controle de disponibilidade do front-end de contingência e feito em paralelo pela tabela CONFIG.

Quando usar o fluxo manual

  • Teste operacional após manutencao.
  • Sincronização pontual de tabela especifica.
  • Diagnostico de divergencia de dados.
  • Validação antes de liberar o servico automatico.

8. Fluxo do Servico Automatico

Implementação atual

O servico foi refatorado para reutilizar exatamente o mesmo fluxo funcional do dashboard manual. Ele nao chama mais metodos legados inexistentes. 

php services/BackupService.php
  -> BackupService
    -> SyncModel
      -> forceTableSync(tabela)
        -> BackupModel::insertDataToLocal(tabela)

Caracteristicas do servico

  • Loop contínuo com retry configuravel.
  • Modo --once para validação controlada.
  • Modo --table=NOME_TABELA para teste de tabela unica.
  • Logs em arquivo logs/sync.log.
  • Instalação como servico Windows via NSSM por install_service.bat.

Exemplos

php services/BackupService.php --once --table=USUARIO
php services/BackupService.php --once
php services/BackupService.php --interval=900 --retry=60

9. Estrategia de Upsert e Commit por Lote

Modelo atual

  • Busca incremental por DT_ATUALIZação ou coluna de controle equivalente.
  • Aplicação de filtros de chave para evitar registros sem identificador útil.
  • Processamento com UPDATE seguido de INSERT quando necessário.
  • Fallback para conflito de unicidade com nova tentativa de UPDATE.
  • Commit por lote no banco local.

Tabelas grandes

Para tabelas de maior volume, como PESSOA_FISICA, COMPL_PESSOA_FISICA e EVOLUCAO_PACIENTE, o sistema usa processamento em lote com leitura em streaming do cursor Oracle. Essa estrategia substituiu a paginação por OFFSET/FETCH, que se mostrou muito pesada para tabelas muito grandes.

Chave composta

COMPL_PESSOA_FISICA usa CD_PESSOA_FISICA + NR_SEQUENCIA como chave de sincronização. O bind OCI foi corrigido para não reaproveitar indevidamente variaveis de loop, evitando falsos ORA-00001.

10. Arquivos Principais

Configuração

  • app/config/database.php: fonte oficial de configuração de conexao e metadados por tabela - CONFIDENCIAL.
  • app/config/databaseConfig.php: intermediário de compatibilidade (shim) para código legado.

Modelos centrais

  • BackupModel.php: sincronização, conexões, retenção, logs e diagnósticos.
  • SyncModel.php: interface operacional para dashboard e serviço.
  • DatabaseConnection.php: helper auxiliar de conexão, reutilizado por pontos legados.
  • UserModel.php: autenticação local baseada em tabela USUARIO.

Scripts auxiliares

  • validate_manual_sync.php: valida o fluxo real por tabela.
  • debug_batch_fetch.php: mede tempo de busca de lote.
  • debug_batch_process.php: testa o processamento local de um lote.
  • debug_constraints.php: inspeciona restricoes locais.

11. Operação e Rotinas

Rotina recomendada de operação

  1. Validar conexoes e ambiente PHP/OCI8.
  2. Executar sincronização manual de tabela quando houver manutencao ou investigação.
  3. Manter o servico automatico em execucao continua para operação normal.
  4. Monitorar logs e tamanho das tabelas rolling.

Comandos uteis

cd /d C:\inetpub\wwwroot\TASYBackup

php validate_manual_sync.php
php validate_manual_sync.php PESSOA_FISICA
php services/BackupService.php --once --table=USUARIO
install_service.bat
net start "TASYBackupService"
net stop "TASYBackupService"

Arquivos de log importantes

  • logs/error.log: logs operacionais e diagnosticos de aplicação.
  • logs/sync.log: logs resumidos do servico.
  • logs/service_stdout.log: saida padrao do processo do servico.
  • logs/service_stderr.log: erros do processo do servico.

13. Solução de Problemas

Servico não inicia com erro especifico 3

  • Causa tipica: AppDirectory invalido no NSSM.
  • Verificar: nssm get TASYBackupService AppDirectory.
  • Corrigir: reinstalar via install_service.bat em prompt elevado como administrador.

ORA-00001 em COMPL_PESSOA_FISICA

  • Historicamente causado por bind incorreto de chave composta.
  • Validar se a configuração de key_columns continua correta.

Sincronização muito lenta em tabelas grandes

  • Verificar se a tabela esta configurada com batch_size.
  • Evitar paginação por OFFSET/FETCH para tabelas muito volumosas.
  • Usar scripts de debug para identificar se o gargalo esta no fetch ou no processamento.

Dashboard funciona e servico não

  • Comparar logs do servico com logs/error.log.
  • Testar primeiro com php services/BackupService.php --once --table=USUARIO.

14. Segurança e Pendências Técnicas

Riscos ainda presentes

  • Atencao Credenciais de banco ainda estao no código em database.php.
  • Atencao Existem scripts auxiliares e testes no raiz que devem ser tratados como utilitarios internos.
  • Atencao O projeto ainda possui pontos legados fora do fluxo principal que merecem padronização futura.

Melhorias recomendadas de segurança

  • Mover credenciais para variaveis de ambiente ou arquivo fora do repositório.
  • Criar usuário Oracle dedicado no XE em vez de operar permanentemente com SYSTEM.
  • Revisar permissões do IIS e do usuário do servico.
  • Restringir exposicao publica de scripts utilitarios e de debug.

15. Recomendações para Evolução do Sistema

  1. Substituir credenciais hardcoded por configuração segura externa ao código.
  2. Padronizar logs em formato estruturado unico.
  3. Implementar health-check de servico e alertas basicos.
  4. Avaliar uso de MERGE em lote no Oracle para tabelas transacionais maiores.
  5. Separar scripts de debug e validação em uma pasta tools/ ou scripts/.
  6. Adicionar documentação complementar para operação do portal ISCMC em contingência.
